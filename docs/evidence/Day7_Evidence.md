## 6. AI 协作复盘

### AI 帮助最大的阶段

1. **Day2—Day3 页面骨架与数据建模**：AI 高效生成页面结构、路由配置和 Mock 数据，大幅减少手工搭建时间
2. **Day4 发布表单**：AI 生成的表单校验逻辑和字段设计提供了良好起点
3. **Day6 状态组件**：LoadingState、EmptyState、ErrorState 的模板代码由 AI 生成，人工调整样式和交互细节
4. **Day7 文档整理**：AI 辅助整理 README、检测报告和证据卡结构

### AI 生成内容中的问题

1. **字段不贴合业务**：AI 初始生成的 Mock 数据字段偏通用化，缺少 campus、status、deadline 等校园特有字段，需要人工逐项补充
2. **过度复杂**：AI 倾向于引入登录认证、图片上传、支付等超出实训范围的功能，需要人为裁剪
3. **组件过度抽象**：部分组件被过度拆分为多层嵌套，反而降低可维护性，需要合并简化
4. **类型不匹配**：`string | number` 与 `number` 类型混用导致 TypeScript 编译错误，需要人工修复
5. **跨 Store 联动缺失**：如购物车与订单的数据联动，AI 倾向于只处理单个 Store 的逻辑

### 我的判断与修改

- 每次 AI 生成代码后，先在浏览器中运行验证，确认功能正常
- 删除超出范围的功能模块，保持项目复杂度可控
- 统一命名规范和数据字段，确保前后一致
- 修复所有 TypeScript 类型错误，确保构建通过
- 简化过度设计的组件结构，降低理解成本

### 哪些内容必须自己理解

- Vue Router 的嵌套路由和导航守卫机制
- Pinia Composition API 的响应式原理
- JSON Server 的 RESTful 约定和查询参数
- TypeScript 类型系统中 `string | number` vs `number` 的区别
- 业务需求的优先级判断和功能取舍

### 以后使用 AI Coding 工具的注意事项

1. 先确认需求边界，再让 AI 生成代码，避免功能膨胀
2. 分步骤验证，不要让 AI 一次性生成过多代码
3. 每次生成后必须人工运行和测试
4. 数据模型设计应前置并与 AI 充分沟通字段含义
5. 保留每次修改的 Git 记录，方便回溯 AI 的改动

## 8. 项目不足与改进方向

| 问题 | 说明 | 改进方向 |
|------|------|----------|
| JSON Server 模拟后端 | 非真实后端，无法处理复杂业务逻辑 | 后续可接入 Node.js/Spring Boot 后端 |
| 模拟登录 | 用户信息存 localStorage，无 JWT 鉴权 | 可接入真实认证服务 |
| 收藏无后端持久化 | 依赖 JSON Server 的 db.json | 接入真实数据库 |
| 无图片上传 | 图片字段仅支持 URL 输入 | 可集成文件上传服务 |
| 搜索较基础 | 纯前端筛选，无模糊搜索 | 后续可接入 Elasticsearch 或后端搜索 |
| Dashboard 包体积大 | ECharts 库体积超 500KB | 使用动态导入按需加载 |
| 移动端适配不足 | 主要为 PC 端设计 | 可增加响应式适配 |

## 9. 今日反思与总结

Day7 的核心价值不在于继续写代码，而是从"完成代码"走向"完成项目"。在综合验收过程中，我深刻体会到：一个实训项目的完成不只是页面能打开，还包括构建是否通过、文档是否完整、过程是否可追溯、AI 协作是否有反思。

通过 7 天的实训，我在以下几个方面有了新的理解：

1. **前端工程化不只是写 Vue 组件**：从需求分析、数据建模、路由设计到状态管理、交互优化、构建部署，每一步都需要工程思维。
2. **AI 是工具而非替代品**：AI Coding 工具可以大幅提高重复性工作的效率（如生成页面骨架、Mock 数据），但业务理解、类型修复、功能取舍和质量判断必须由开发者自己完成。本次实训真正训练的，是"管理 AI、审查 AI、修正 AI"的能力。
3. **版本管理是过程证据的核心**：Git 提交不仅是代码备份，更是开发过程的完整记录。每天及时提交、写好提交信息，是工程规范的基本要求。
4. **证据卡不是形式主义**：记录每日工作、遇到的问题和解决方法，有助于形成完整的学习闭环，也为项目展示和答辩提供了扎实素材。
5. **"完成"比"完美"更重要**：Day7 不追求堆砌新功能，而是保证已有功能稳定可演示、文档完整、提交规范。这是一种务实的工程态度。
