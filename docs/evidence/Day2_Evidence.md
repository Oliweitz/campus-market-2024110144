Day 2 — 页面骨架、路由导航与多业务场景扩展

今天的目标是在 Vue3 项目基础上扩展多页面结构、完善路由、构建四分类业务场景的完整前端框架。

首先是骨架搭建。我在 src/views 下新增了 ListView、DetailView、PublishView、MessageView、ProfileView、DashboardView 六个页面，加上已有的 HomeView 共七个。路由在 router/index.ts 中配了七条，Detail 路由用 :id 支持动态参数。App.vue 的导航栏放了首页、列表、发布、消息、我的、看板六个入口，标题点击也可以跳转看板。

然后是数据层重构。我把四个业务场景的数据统一放到了 data/listings.ts，定义了 Listing 接口，包含类型、校区、状态、发布时间、标签等公共字段，以及每种类型的差异化字段。一共 13 条种子数据。后续又建了 listings Store 把数据变成响应式，这样拼单人数变化、发布新信息、状态切换都能实时反映到所有页面。

列表页做了下拉筛选，支持分类、校区、状态三个维度，还有关键词搜索框。每一项右边有收藏按钮，收藏状态通过 favorites Store 全局同步。详情页根据四种类型渲染不同字段，发布者名字点击可跳转消息页。

购物车和订单系统是今天交互最复杂的部分。cart Store 管理加入/删除/购买，购买时联动 orders Store 记录订单、联动 listings Store 扣减库存，库存归零自动将商品状态设为已完成并撤出活跃列表。个人中心新增了订单列表，展示每次购买的商品、数量和累计消费。

拼单和跑腿的参与/退出逻辑花了不少时间。参与拼单后人数加一，满员自动完成；退出时人数减一，不满则恢复为进行中。跑腿接单后直接从列表撤下（状态变已完成），退出后重新回到列表。详情页根据当前用户是否为发布者展示两套按钮——发布者看到撤回/重新发布/已找到失主，浏览者看到加入购物车/参与拼单/接单。

消息页从空白开始，只有通过详情页点击用户名才会动态添加联系人。发送消息后 800ms 自动生成模拟回复，未读消息数实时显示在联系人列表和导航栏上。

发布页做了四种类型的动态表单——选择二手交易就显示价格、库存、成色；选择失物招领就显示丢失/拾获和日期；拼单显示目标人数和截止时间；跑腿显示酬劳和任务地点。提交后写入 listings Store 并自动跳转到新详情页。

看板页的数据从 listings Store 实时计算，用 CSS 柱状图展示类型占比和校区分布。

今天遇到的一个 bug 是 ProfileView 里 @click 写了 JS 块注释导致 Vue 模板编译失败。查了才发现 vue-tsc 只检查 TypeScript，模板语法错误要到 pnpm build-only 才会暴露，以后验证要以构建为准。

最后是实验思考。第一个问题，为什么要把数据放在共享模块而不是每个页面写一份？因为同一个数据在列表、详情、看板、个人中心都要用，分开写会导致数据不一致，改一处漏一处。第二个问题，响应式数据为什么重要？如果用普通数组，拼单人数变了、发布新信息了，其他页面不会自动刷新。Pinia Store 的存在就是为了让所有组件共享同一份响应式数据。第三个问题，状态流转（active/completed/closed）为什么要设计三个状态而不是两个？因为撤回（closed）和完成（completed）是不同的概念——撤回可以重新发布，完成了就不行。如果只用一个"结束"状态，就区分不了这两种情况。
