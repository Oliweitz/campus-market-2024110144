Day 1 — 需求分析、架构理解与项目初始化

日期：2026年6月27日

今天我第一次接触到校园轻集市这个项目。它是一个面向高校的二手闲置交易前端应用，主要解决校园里闲置物品的流转问题——比如毕业季学长学姐出旧教材、闲置的电子产品这些。核心的业务场景大致包括商品浏览、商品发布、商品详情、搜索筛选还有用户系统。七天的实训目标就是把这个 Vue 3 前端项目完整地做出来，中间每一步都按软件工程的流程走。

接下来我花了不少时间看项目目录。src 下面是自己要动手写的部分，分成 api、components、router、stores、views 这几个目录，每个各司其职。api 放接口请求，components 放公共组件，router 管页面路由，stores 用 Pinia 做状态管理，views 放每个页面的视图。docs 和 scripts 是课程给的基础设施，不需要自己动。

然后我把三个核心文件读了一遍。main.ts 是项目的入口，做的事情很清晰——创建 Vue 应用，把 Pinia 和 Router 注册进去，最后挂载到页面上。router 的 index.ts 目前只配了首页这一条路由，之后每加一个新页面就要在这补一条。App.vue 是整个应用的根组件，里面有个 RouterView，它决定了当前路由对应的页面渲染在哪。

AI 协作这块，我今天主要用 Claude 帮了两件事。第一是问清楚这个项目是干什么的，它很快帮我理出了业务背景和技术栈。第二是让它按实验手册的要求帮我搭了今天这份证据卡的框架。不过内容我还是自己审核修改了一遍，毕竟 AI 的输出不能直接当最终结果用。

关于接下来的开发规划，我列了六个页面，按优先级排了一下。首页的商品列表和商品详情是 P0，最核心；发布商品、登录和注册是 P1；个人中心放最后。开发顺序也按这个优先级来，先把核心的浏览和详情跑通，再补用户和发布功能。

今天遇到的一个小问题是 Git 提交时报了身份不明的错，原因是仓库没有配 user.name 和 user.email。查了一下才知道这两个值要和自己 GitHub 账号的信息对应上，配好之后就正常提交了。

最后是今天的实验思考。第一个问题，为什么第一天不直接写页面？我觉得是因为如果连项目结构和业务场景都没搞懂就开始敲代码，方向很容易偏，后面大概率要返工。第二个问题，理解项目为什么比编码更重要——这其实是软件工程的老道理了，需求理解和架构设计出问题，代码写得再漂亮也没用。第三个问题，AI 应该承担什么角色？我的看法是它是个很厉害的辅助工具，能解释代码、生成模板、帮着排查问题，但关键的判断和决策还得自己来。今天最大的收获就是从头理解了 Vue 3 工程是怎么组织起来的，之前看项目目录像黑盒，现在至少知道每个文件夹在干什么了。
