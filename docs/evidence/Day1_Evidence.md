## 1. 今日完成内容

## 2. 项目架构设计

| 架构层次 | 目录/文件 | 职责 |
|---|---|---|
| 视图层 | `src/views/` | 页面级组件，每个路由对应一个视图 |
| 组件层 | `src/components/` | 可复用的 UI 组件（卡片、表单、导航等） |
| 状态层 | `src/stores/` | Pinia 状态管理，跨组件共享数据 |
| 接口层 | `src/api/` | Axios 请求封装，与后端/Mock 通信 |
| 路由层 | `src/router/` | Vue Router 配置，页面跳转与导航守卫 |
| 入口层 | `main.ts` + `App.vue` | 应用初始化、插件注册、根布局 |

技术栈选型为 Vue3 + Vite + TypeScript + Vue Router + Pinia + Axios + JSON Server。TypeScript 提供类型安全，Pinia 替代 Vuex 做轻量状态管理，JSON Server 零配置提供 Mock API。

## 3. 页面规划与优先级

| 页面 | 优先级 | 路由 | 说明 |
|---|---|---|---|
| 首页/商品列表 | P0 | `/` 或 `/home` | 核心浏览入口，最先实现 |
| 商品详情 | P0 | `/detail/:id` | 从列表进入，展示完整信息 |
| 发布商品 | P1 | `/publish` | 数据入口，依赖列表页就绪 |
| 登录/注册 | P1 | `/login` `/register` | 用户体系基础 |
| 消息 | P2 | `/message` | 依赖用户和商品数据 |
| 个人中心 | P2 | `/user` | 汇总展示，最后集成 |

优先级依据：先让用户能"看"（浏览和详情），再让用户能"发"（发布），最后做"管"（消息和个人中心）。

## 8. 今日反思

第一天不直接写代码而是先理解项目结构和业务场景，这本身就是一个重要的工程实践。如果连项目是怎么组织起来的、每个文件在干什么都没搞懂就开始敲代码，方向很容易偏，后面大概率要返工。需求理解和架构设计出问题，代码写得再漂亮也没用。

关于 AI 的角色定位：AI 是很好的辅助工具，能快速解释代码结构、生成文档框架、回答技术问题。但关键的判断——比如哪些页面先做、技术栈为什么这样选、目录结构是否合理——必须由开发者自己做出。AI 可以给建议，但决策权和责任都在开发者身上。今天最大的收获是从头理解了 Vue3 工程化项目的组织方式，之前看项目目录像黑盒，现在至少知道每个文件夹在干什么、数据在各层之间是怎么流动的。
