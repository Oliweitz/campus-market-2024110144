# Day4 Evidence — 发布表单与数据新增

## 1. 今日完成内容

完成了校园轻集市的发布表单设计与数据新增功能。实现了四种业务类型（二手交易、失物招领、拼单搭子、跑腿委托）的类型切换表单，包含通用字段和专属字段。完成了基础表单校验（必填项校验、数值范围校验），通过 Pinia Store + Axios 向 JSON Server 发送 POST 请求实现数据新增，提交成功后自动跳转到对应列表页并给出用户反馈。表单切换类型时自动重置，避免字段残留。

## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---|---|---|---|
| 二手交易 | items (type=secondhand) | title、category、price、condition、location、description、stock、allowBargain | 需要展示商品分类、价格、成色和交易条件 |
| 失物招领 | items (type=lostfound) | title、lostOrFound、itemName、location、eventTime、itemFeature、contact、description | 需要区分寻物/招领，描述物品名称、特征和联系方式 |
| 拼单搭子 | items (type=group) | title、targetCount、deadline、location、description | 需要说明人数目标和截止时间 |
| 跑腿委托 | items (type=errand) | title、taskType、reward、from、to、taskPlace、deadline、expectedTime、description | 需要说明任务类型、取送地点、酬劳和时限 |

字段设计遵循"通用字段共享 + 专属字段按类型展示"的原则。通用字段包括标题、校区、地点、描述、标签、图片。专属字段只在对应发布类型下展示，避免表单冗余。

## 3. 表单校验规则

本日实现的表单校验以基础必填和简单数值判断为主，未引入第三方校验库：

- **通用字段**：标题不能为空、地点不能为空、描述不能为空
- **二手交易**：商品分类不能为空、价格必须大于 0、成色必须选择
- **失物招领**：物品名称不能为空、发生时间必须选择
- **拼单搭子**：目标人数不能少于 2 人、截止时间必须选择
- **跑腿委托**：任务类型不能为空、酬劳不能为负数、取件地点不能为空、送达地点不能为空、任务地点不能为空、截止时间必须选择

校验函数 `validateForm()` 先清空所有错误信息，逐字段检查后将错误写入 `errors` 响应式对象，最终返回是否全部通过。校验未通过时阻止表单提交，错误信息实时显示在对应表单项下方。

## 7. 遇到的问题与解决方法

| 问题 | 原因 | 解决方法 |
|---|---|---|
| 表单提交后 `db.json` 中新增记录缺少 category、itemName 等字段 | AI 生成的代码未包含规范要求的部分业务字段 | 手动在 PublishForm.vue 中补充 ref 变量、校验规则、重置逻辑、提交映射和模板 |
| 跑腿委托缺少取件/送达地点的区分 | 原设计只有单一 taskPlace 字段 | 新增 from（取件地点）和 to（送达地点）两个独立字段，并排展示 |
| TypeScript 类型检查报错 | Item 接口中未定义新增字段 | 在 listings.ts 的 Item 接口中为各业务类型补充可选字段 |

## 8. 今日反思

发布表单、表单校验和数据新增是 Web 应用从"只读展示"走向"用户交互"的关键一步。发布表单不仅仅是堆几个输入框，它涉及业务类型判断（该收集什么字段）、字段设计（字段是否贴合业务场景）、表单校验（哪些输入是必须的、哪些需要格式检查）、数据提交（如何将用户输入转化为结构化数据并发送到正确的接口）、成功反馈（让用户知道操作结果）和页面跳转（引导用户到合理的目标页面）等多个环节。

通过本日实训，我理解了：前端的核心价值不仅是展示后端数据，更在于收集用户输入并将其可靠地转化为业务数据。如果发布流程设计不清（字段缺失、校验遗漏、提交路径错误），后续的用户状态管理、我的发布记录、消息中心等功能都会失去数据基础。表单校验看似简单，但它直接关系到数据质量和用户体验——让用户尽早发现问题，而不是提交后才发现错误。数据新增是整个应用的"数据入口"，入口设计得好，后续功能才能建立在可靠的数据之上。
