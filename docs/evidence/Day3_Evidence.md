# Day3 Evidence — Mock 数据建模与列表渲染

## 1. 今天做了什么

今天是项目变化最大的一天。之前两天搭好了页面骨架和路由，但所有页面都是静态的，列表里显示的是写死在模板里的占位文字。今天的目标很明确：让页面从「死」的变成「活」的——也就是让数据驱动页面。

上午主要在做数据建模。校园轻集市有四类业务：二手交易、失物招领、拼单搭子、跑腿委托。每类业务需要什么字段？字段之间有什么关系？这些问题是动手写代码之前必须想清楚的。我在 `db.json` 里设计了五组数据集合：`trades` 放二手商品，`lostFounds` 放失物招领，`groupBuys` 放拼单搭子，`errands` 放跑腿委托，再加上 `users` 放用户信息。每组准备了五六条贴近校园生活的示例数据——教材转卖、捡到耳机、奶茶拼单、代取快递之类，都是大学生日常会遇到的事。

下午安装了 JSON Server，在 `package.json` 里加了 `mock` 启动命令。JSON Server 的好处是只需要一个 `db.json` 文件，它就能自动生成完整的 RESTful 接口——`GET /trades` 返回商品列表，`GET /trades/1` 返回单条详情。对教学项目来说，这种方式比搭建真实后端合适得多：能体验前后端分离的完整流程，又不用碰数据库和服务器。

接着做 API 封装。在 `src/api` 下面创建了 Axios 实例文件 `http.ts`，然后按业务域拆了四个模块——`trade.ts`、`lostFound.ts`、`groupBuy.ts`、`errand.ts`，每个模块里定义了 TypeScript 接口和请求方法。这样做的好处是页面组件不需要直接写请求地址和参数，调用 `getTrades()` 就行，后面如果要换接口路径，改一个文件就够了。

页面渲染是今天最见效果的部分。之前 TradeView 里是空的，现在打开页面就能看到一排商品卡片，标题、价格、成色、地点都有。四个业务页面都接入了数据——TradeView 展示二手商品列表，LostFoundView 区分丢失和拾获，GroupBuyView 显示拼单进度条，ErrandView 标注酬劳和路线。首页的四个分类卡片也加上了跳转链接，点「二手交易」就去 `/trade`。

还做了一个重要的基础设施：用户认证。给 `db.json` 里三个预设用户配了统一的账号——张三用 `zhangsan` 登录，李四用 `lisi`，王五用 `wangwu`，密码都是 `123456`，方便演示。创建了登录页和注册页，注册时会检查用户名有没有被占用。登录状态存在 localStorage 里，刷新页面不会掉。发布页和消息页加了登录保护，没登录的话会显示引导提示而不是报错。

最后做了数据持久化。购物车、订单记录、拼单和跑腿的参与记录，这些原本只是 Pinia 内存里的数据，一刷新就没了。我用 `watch` 监听这些 Store 的变化，每次数据变更自动写入 localStorage，初始化时再从 localStorage 恢复。这样用户的操作不会因为刷新而丢失。

## 2. 数据字段是怎么设计的

设计数据字段这件事，一开始觉得就是列几个属性，但真正动手才发现每个字段背后都有一个业务问题要回答。

二手交易的核心是价格和成色。买家最关心两件事：这东西多少钱？新旧程度怎么样？所以 `price` 和 `condition` 是必不可少的。`category` 用来分类浏览——是教材还是数码还是生活用品——这样用户可以按类别筛选。`location` 是交易地点，校园里面对面交易，知道在哪碰头很重要。

失物招领跟二手交易完全不同。它不涉及价格，核心是帮失主和拾到者对接。`type` 字段用来区分「丢失」和「拾获」——这两个场景虽然都在同一个页面上，但信息的方向是相反的。`itemName` 是物品的名称，`eventTime` 是丢失或捡到的时间点，`itemFeature` 描述物品的特征——颜色、大小、特殊标记，这些是认领的关键线索。

拼单搭子有意思的地方在于它是一个「进行中」的状态。`targetCount` 是目标人数，`currentCount` 是当前已有的人数，两个一除就是拼单进度。`deadline` 是截止时间，过了这个时间还没凑够人的话，拼单就失效了。这几个字段组合在一起，用户一看就知道还能不能参与。

跑腿委托更像是发布一个小任务。`reward` 是酬劳，`from` 和 `to` 描述了取送的路线——从菜鸟驿站到南区宿舍，从打印店到教务办公室。对想接单的人来说，路线顺不顺路、酬劳合不合理，是最直接的判断依据。`taskType` 用来区分任务类型——取快递、代买、打印、搬运，不同的任务对体力和时间的要求不一样。

用户体系的设计走了最简单的路线。姓名拼音当用户名，密码统一 `123456`。这不是偷懒——对教学演示项目来说，复杂的注册流程只会增加无关的认知负担。需要关注的是注册时自动查重防止用户名冲突、登录态持久化保证刷新不掉、以及信用分的教学展示意义。

## 3. 我的设计思路

数据建模这件事，实训手册上给了参考字段，AI 也能生成一份挺完整的数据结构。但真正决定用哪些字段、怎么命名、值怎么取，还是需要自己想清楚。

我的出发点一直是一个很简单的标准：打开这个页面的人，看到这行数据，能不能做出判断？比如看到一条二手商品，有价格、有成色、有地点，就能决定要不要联系卖家。看到一条拼单信息，有目标人数、有当前人数、有截止时间，就能判断还能不能参与。如果一个字段对决策没有帮助，那它就不是必要的——至少在这个阶段不是。

命名上我力求统一。最初 AI 生成的字段叫什么的都有——`createTime`、`postTime`、`publishTime`——但含义其实都一样。我全部统一成了 `createdAt` 和 `updatedAt`，跟后端惯例保持一致。状态值也做了简化：`open`、`closed`、`done` 三个值覆盖了所有场景，比 AI 最初生成的五花八门的状态名清晰得多。

持久化策略是我比较满意的一个设计决策。我没有把所有数据都往 Store 里塞，也没有把所有数据都扔给 JSON Server。我划了三条线：JSON Server 管业务数据——商品、收藏、消息，这些是需要跨会话持久化的核心资源；localStorage 管客户端状态——购物车、登录态、参与记录，这些是跟当前用户绑定但不属于核心业务的数据；Pinia 管运行时状态——当前列表、筛选条件、加载状态，这些是页面运行期间需要跨组件共享但不需要持久化的。三条线各司其职，互不越界。

## 4. AI 帮了什么忙，又踩了什么坑

今天的绝大部分代码都有 AI 参与。从 `db.json` 的数据结构到四个 API 模块的 TypeScript 接口，从 `ItemCard.vue` 的组件模板到四个业务页面的完整代码，从登录注册的表单到 userStore 的登录逻辑，AI 几乎是全流程参与。

AI 做得好的地方很明显。TypeScript 类型定义一生成几乎没有类型错误——这在手写代码时往往要反复调试。代码结构也很清晰，API 模块、组件、Store 的职责分离做得到位，不会把所有东西搅在一起。注册时自动检查用户名是否已存在这个细节，AI 是主动加的，说明它考虑到了完整流程而不仅仅是单个接口调用。

但 AI 也出了不少需要我花时间排查的问题。最头疼的一个是 `<template v-else>` 里面嵌套 `<template v-if>`——这种写法 Vue 模板编译器根本不认，导致整个页面加载不出来。花了好一阵子才定位到问题，最后把条件判断都移到普通 `<div>` 上才解决。这说明模板语法的边界条件，AI 的理解是有盲区的。

另一个典型问题是 AI 倾向于「毁了就重来」而不是「坏了就修」。userStore 的 `init()` 方法在 API 请求失败时会直接调用 `logout()` 清除登录态——这意味着 JSON Server 如果暂时不可用，用户就会被踢出去。我改成了静默忽略错误，保留本地缓存，后台再重试。产品思维上的这种容错意识，AI 目前还不太具备。

还有数据内容的问题。AI 生成的 Mock 数据有一些很「通用」——手机壳、充电宝之类的，放在任何电商平台上都行，但缺乏校园场景的辨识度。我把数据内容从头到尾替换了一遍，全部改成校园里真实会出现的事情：高数教材转卖、图书馆丢钱包、周末火锅拼单、代取快递送到宿舍楼下。数据内容贴不贴场景，看的人一眼就能感觉到。

## 5. 具体改了 AI 哪些东西

第一件事是替换 Mock 数据内容。AI 生成的数据太泛了，像是从通用电商模板里抽出来的。我一条条改成了校园场景：机械键盘、折叠书桌、LED 台灯、捷安特自行车、四六级真题集、寻找钱包和 U 盘、奶茶火锅拼单、羽毛球搭子、代取快递、搬宿舍行李。这些内容一看就是校园生活。

第二件事是解决了数据不一致的问题。最初 AI 设计了两套并行的数据模型——四个独立集合（`trades` 等）和统一 `items` 集合。四个新页面各自请求各自的独立集合，列表页请求统一集合，两边虽然是同一批商品的数据镜像，但字段结构不同，ID 映射也不完全对齐。我把四个页面全部改为从 Pinia 的 `itemStore` 获取数据，利用 `computed` 自动响应，发布新信息后所有页面自动同步，不再需要手动刷新。

第三件事是字段命名统一。AI 在不同的文件里用了不同的时间字段名——`createTime`、`publishTime`、`postTime`，但含义都是发布时间。我统一成了 `createdAt` 和 `updatedAt`，状态值统一用 `进行中 / 已完成 / 已关闭` 三个中文值。

第四件事是修复了好几处功能性 bug。发布页和消息页在未登录状态下会因为 `currentUserId` 为 null 而报错——我加了登录引导。userStore 的 `init()` 在 API 失败时不应该清除登录态——我改成了静默容错。拼单页面缺少进度条的可视化展示——我补了一个简单的进度条组件。

第五件事是样式对齐。AI 生成的组件里大量使用了硬编码的颜色值——`#2563eb`、`#ef4444`、`#6b7280` 等散落在各处。我把这些全部替换为项目的 CSS 变量——`var(--primary)`、`var(--danger)`、`var(--text-light)`——这样以后如果要换主题色，改一个地方就够了。

第六件事是持久化补全。购物车、订单、参与记录这三个 Store，AI 最初的实现是纯内存状态，刷新就没了。我给每个 Store 加了 `watch` 监听，数据变更时自动 `JSON.stringify` 写入 localStorage，初始化时再读回来。用户操作不会因为误刷新而前功尽弃。

## 6. 踩过的坑

第一个坑是数据不一致。四个新页面的数据来自四个独立集合，列表页的数据来自 `items` 统一集合。虽然内容是我手动对齐的，但字段结构不同——独立集合用 `publishTime`，统一集合用 `createdAt`；独立集合用 `publisher` 字符串，统一集合用 `publisherId` 数字。两边对不上的时候，详情页可能拿到的是独立集合的数据结构，列表页拿到的是统一集合的结构，同一个商品在不同页面上显示的信息不一样。最终我把所有视图都接入 Pinia store，从 `items` 单一数据源取值，才彻底解决了这个问题。

第二个坑是刷新后登录态丢失。userStore 的 `init()` 方法调用 `userApi.getUserById()` 获取完整用户信息，如果 API 请求失败——比如 JSON Server 还没启动——就直接 `logout()` 清除 localStorage。这意味着每次重启服务，用户都会被踢出去重新登录。我把 `init()` 改成 API 失败时不清除登录态，只静默忽略，同时把昵称和校区缓存到一个独立的 localStorage key 里，刷新后先用缓存数据撑起 UI，后台再异步刷新完整数据。这样即使 JSON Server 暂时不可用，用户也感觉不到。

第三个坑是 Vue 模板编译错误。AI 在一个 `<template v-else>` 里嵌套了多层的 `<template v-if>`，这种写法 Vue 的模板编译器不认，导致整个页面的组件都加载失败，路由跳转也被阻断。问题出在 Vue 的 `<template>` 标签上不能同时使用 `v-else` 和内部的 `v-if`。把条件判断移到普通 `<div>` 上就解决了。这种模板编译器的边界行为，AI 目前的训练数据里似乎覆盖得不够。

第四个坑是购物车数据刷新丢失。cart、orders、mylist 三个 Store 一开始都是纯 Pinia 内存状态，没有任何持久化。用户加了几件商品到购物车，不小心刷新了页面，全没了。我的解决方案是给每个 Store 加 `watch` 自动持久化到 localStorage，初始化时恢复。代码不多，但影响很大。

## 7. 今天的感受

今天是真正开始「写一个完整应用」而不是「写几个页面」的一天。之前两天做骨架和路由的时候，感觉自己更像在搭积木——页面有了，导航通了，但里面是空的。今天把数据接进去之后，项目才第一次有了「应用」的感觉——打开二手交易页面能看到真实的商品列表，下拉筛选能过滤内容，点进去能看到详情。

Mock 数据建模这件事给我的最大启发是：前端不是等后端给什么就用什么，而是要先想清楚自己要什么数据、怎么组织这些数据、每个字段服务于什么功能。JSON Server 给了我们一个不依赖后端的验证环境，但数据的质量——贴不贴业务、全不全面、好不好用——完全取决于前端开发者自己的判断。

最有收获的是踩坑和修坑的过程。Vue 模板编译错误的排查让我更理解了模板语法的限制，登录态持久化的修复让我意识到容错设计的重要性，数据一致性问题的解决让我重新思考了单一数据源的价值。这些都不是看教程能学到的，只能靠实际踩坑积累。

今天写了不少代码，但印象最深的不是代码本身，而是在几个方案之间反复比较、最终选定一个然后执行到底的过程。独立集合还是统一集合？API 模块怎么拆分？Store 存什么、localStorage 存什么？这些问题没有标准答案，但一旦决定了就要贯彻——最怕的就是两头都留着，变成一个谁也不敢动的烂摊子。

## 8. 后续的收敛

Day3 之后项目又经历了一轮架构整理。最主要的变化是放弃了最初的两套数据模型并存方案，彻底统一到 `items` 单一集合上。四个独立集合——`trades`、`lostFounds`、`groupBuys`、`errands`——被从 `db.json` 里删除了，连带四个从来没用过的 API 文件也一并清理。全项目现在只通过 `items` 表的 `type` 字段区分四类业务，一个端点、一套字段、一种数据格式。

命名也做了一次全面对齐。对照实训手册，`request.ts` 改成了 `http.ts`，`userStore.ts` 和 `favoriteStore.ts` 改成了 `user.ts` 和 `favorite.ts`，个人中心页从 `ProfileView.vue` 改成了 `UserCenterView.vue`，路由路径也修正了——`/lostfound` 变成了 `/lost-found`，`/groupbuy` 变成了 `/group-buy`。名字看起来是小事，但一个项目里名称不一致，就像一本书的前后半部分用了不同的术语，阅读体验会很差。

App.vue 做了瘦身。头部和导航栏被抽取成了 `AppHeader.vue` 和 `AppNav.vue` 两个独立组件，又加了一个 `AppLayout.vue` 作为页面布局容器。这样 App.vue 的行数从两百多行降到了九十行左右，打开文件一眼就能看到整体结构，不用在一堆 CSS 里找逻辑。

还补了几个之前遗漏的工具组件。`FormField.vue` 封装了表单项的 label、必填星号和错误提示，后面发布表单直接就用了。`LoadingState.vue` 和 `ErrorState.vue` 分别处理加载中和请求失败两种状态，`SearchBar.vue` 封装了关键词输入和清空功能。这些组件都很小——每个不过几十行——但有了它们之后，页面代码不用重复写同样的状态判断了。

整套收敛做下来，最深的感触是：代码不是写得越多越好，而是越少越好——在功能完整的前提下，文件少、集合少、命名一致、结构清晰，才是好的工程状态。Day3 的探索和试错是必要的，但一旦找到了更好的方案，就应该果断执行，把旧的痕迹清理干净。
